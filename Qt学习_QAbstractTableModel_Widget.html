Qt Model/View 框架 · 自定义数据模型核心基类

Qt 6.8 Qt Core / Model-View 2026-06-12

为什么要自定义 Model？

Qt 内置了 QStandardItemModel，但它有几个痛点：

痛点QStandardItemModel自定义 Model（继承 QAbstractXxxModel） 内存开销每个单元格一个 QStandardItem 对象直接映射底层数据结构，零额外对象 数据同步需要手动双向同步（数据源 ⇄ Model）data() 直接读取源数据，数据源即 Model 性能大数据量（10w+行）插入/删除慢beginInsertRows 批量操作，O(1) 通知 类型安全QVariant 满天飞你的数据结构是什么类型，Model 就用什么类型

继承层次

QAbstractItemModel              // 最顶层抽象，提供树形模型能力
  ├── QAbstractListModel        // 一维列表（推荐用于列表类视图）
  └── QAbstractTableModel      // 二维表格（推荐用于表格类视图）
💡 选择建议
列表视图（QListView、QComboBox）→ 继承 QAbstractListModel
表格视图（QTableView）→ 继承 QAbstractTableModel
树形视图（QTreeView）→ 继承 QAbstractItemModel
两者都可以用于 QTableView（底层都是 QAbstractItemModel），但继承专用子类可以减少需要实现的虚函数数量。

QAbstractTableModel — 必须实现的 3 个虚函数

class EmployeeModel : public QAbstractTableModel
{
    Q_OBJECT
public:
    // ① 行数 — 必须
    int rowCount(const QModelIndex &parent = QModelIndex()) const override {
        if (parent.isValid()) return 0;   // 顶层：没有父项
        return m_employees.size();             // 返回数据行数
    }

    // ② 列数 — 必须
    int columnCount(const QModelIndex &parent = QModelIndex()) const override {
        if (parent.isValid()) return 0;
        return 5;  // 姓名/部门/职位/薪资/入职日期
    }

    // ③ 数据访问 — 必须
    QVariant data(const QModelIndex &index,
                  int role = Qt::DisplayRole) const override {
        if (!index.isValid()) return {};
        const auto &emp = m_employees[index.row()];

        if (role == Qt::DisplayRole) {
            switch (index.column()) {
                case 0: return emp.name;
                case 1: return emp.department;
                case 2: return emp.title;
                case 3: return emp.salary;
                case 4: return emp.hireDate;
            }
        }
        return {};
    }

private:
    QVector<Employee> m_employees;  // ← 你的真实数据
};

QAbstractListModel — 只需 2 个虚函数

class TodoListModel : public QAbstractListModel
{
    Q_OBJECT
public:
    int rowCount(const QModelIndex &parent = QModelIndex()) const override {
        return parent.isValid() ? 0 : m_todos.size();
    }                                    // ↑ 不需要 columnCount!

    QVariant data(const QModelIndex &index,
                  int role = Qt::DisplayRole) const override {
        if (!index.isValid()) return {};
        if (role == Qt::DisplayRole)
            return m_todos[index.row()].title;
        if (role == Qt::CheckStateRole)
            return m_todos[index.row()].done
                   ? Qt::Checked : Qt::Unchecked;
        return {};
    }

private:
    QVector<TodoItem> m_todos;
};
🎯 Role 体系 — data() 的核心机制
View 通过不同 role 向 Model 请求不同维度的数据。常用 role： Qt::DisplayRole（显示文本）、Qt::DecorationRole（图标）、
Qt::ToolTipRole（提示）、Qt::TextAlignmentRole（对齐）、
Qt::ForegroundRole（前景色）、Qt::BackgroundRole（背景色）、
Qt::FontRole（字体）、Qt::CheckStateRole（勾选框）、
Qt::UserRole（自定义数据，从这里起步）

让 Model 可编辑 — 需要重写的虚函数

// ① 告诉 View 哪些单元格可编辑、可选、可拖拽
Qt::ItemFlags flags(const QModelIndex &index) const override {
    if (!index.isValid()) return Qt::NoItemFlags;
    return Qt::ItemIsSelectable | Qt::ItemIsEnabled
           | Qt::ItemIsEditable;  // ← 加上这个才是可编辑
}

// ② View 编辑完成后，把新值写回 Model
bool setData(const QModelIndex &index,
               const QVariant &value,
               int role = Qt::EditRole) override {
    if (!index.isValid() || role != Qt::EditRole)
        return false;

    auto &emp = m_employees[index.row()];
    switch (index.column()) {
        case 0: emp.name = value.toString(); break;
        case 1: emp.department = value.toString(); break;
        // ...
    }
    emit dataChanged(index, index, {role});  // 必须发射!
    return true;
}

// ③ （可选）提供表头文本
QVariant headerData(int section, Qt::Orientation orientation,
                    int role = Qt::DisplayRole) const override {
    if (role != Qt::DisplayRole) return {};
    if (orientation == Qt::Horizontal) {
        switch (section) {
            case 0: return "姓名";
            case 1: return "部门";
            // ...
        }
    }
    return {};
}

在线演示 — 可编辑表格

⚠️ setData 后必须发射 dataChanged
如果你在 setData() 里修改了数据但没有调用 emit dataChanged(index, index)， View 不会自动刷新！这是最常见的 bug 之一。
如果修改涉及多行/多列，使用 dataChanged(topLeft, bottomRight) 一次性通知矩形区域。

核心概念：Model 通知 View 数据已变更

六大数据变更通知模式

操作成对方法说明
插入行 beginInsertRows(parent, first, last)
endInsertRows()		 必须先修改底层数据，再发通知
删除行 beginRemoveRows(parent, first, last)
endRemoveRows()		 先通知，再删数据
移动行 (Qt 6) beginMoveRows(srcParent, srcFirst, srcLast, dstParent, dstRow)
endMoveRows()		 Qt 6 新增，Qt 5 需手动 remove + insert
单元格更新 emit dataChanged(topLeft, bottomRight, roles) 不需要 begin/end 配对
列操作 beginInsertColumns / endInsertColumns
beginRemoveColumns / endRemoveColumns		 表格模型专用
全局重置 beginResetModel() / endResetModel() 大量数据变更时最简单的方式

实战示例 — 添加行 + 删除行

void EmployeeModel::addEmployee(const Employee &emp) {
    int newRow = m_employees.size();     // 新行索引
    beginInsertRows(QModelIndex(), newRow, newRow); // 通知 View: 我要在第 newRow 行插入
    m_employees.append(emp);                     // ← 在这之间修改数据!
    endInsertRows();                              // 通知 View: 我插完了，刷新吧
}

void EmployeeModel::removeEmployee(int row) {
    if (row < 0 || row >= m_employees.size()) return;
    beginRemoveRows(QModelIndex(), row, row);
    m_employees.removeAt(row);                   // ← 在这之间删除数据!
    endRemoveRows();
}

// 批量更新单格
void EmployeeModel::refreshRow(int row) {
    auto topLeft = index(row, 0);
    auto bottomRight = index(row, columnCount() - 1);
    emit dataChanged(topLeft, bottomRight);  // 整行刷新
}
🔥 begin/end 必须严格配对！
如果调用了 beginInsertRows 但忘了调用 endInsertRows（或者中间抛异常）， Model 会进入不一致状态，导致 View 崩溃或无法正常显示。
强烈推荐用 RAII 包装器或在 try-catch 中确保 end 一定会被调用。

🔟 条核心避坑要点

① parent.isValid() 必须返回 0
TableModel 和 ListModel 是扁平模型，没有任何层级。当你收到 parent.isValid() == true 的请求时，直接返回 0 —— 因为你的数据没有子节点。
② 永远不要存储 QModelIndex
QModelIndex 是瞬态对象。每次数据变更后，旧的 QModelIndex 可能失效。
如果你需要跨操作记住某个位置，存储 数据本身的 ID/主键，而不是 QModelIndex。
③ data() 中 index.isValid() 检查是必须的
View 可能在各种边界情况下传递无效 index（如 -1 行），你必须在 data() 最前面检查 index.isValid()。
④ 不要在 data() 中做耗时操作
data() 会被频繁调用（每次重绘、每次滚动）。如果里面有数据库查询、网络请求、复杂计算，View 会严重卡顿。
解决方案：缓存计算结果、预加载数据、使用 canFetchMore() 懒加载。
⑤ setData 后忘记 emit dataChanged
View 不会自动感知内存数据的变化。必须显式发射 dataChanged 信号。
⑥ beginInsertRows 的参数含义
beginInsertRows(parent, first, last) 中 first 是插入后第一行的索引，last 是插入后最后一行的索引。
例如要在末尾插入 1 行，当前有 10 行：beginInsertRows(QModelIndex(), 10, 10)，不是 9！
⑦ begin/end 之间不要发射信号或处理事件
在 beginInsertRows 和 endInsertRows 之间只修改数据，不要调用 processEvents()， 不要发射其他可能触发 View 更新的信号，否则会导致未定义行为。
⑧ 线程安全
Model 的所有方法都必须在 GUI 线程（主线程） 中调用。如果数据来自工作线程， 在工作线程中准备好数据后，通过信号槽将数据传回主线程，再由主线程调用 begin/end 系列方法。
⑨ beginResetModel 是最简单的刷新方式
当你不确定数据变了多少、或者需要整体替换时，用 beginResetModel() / endResetModel() 最安全。
代价：View 会丢失所有选中状态、滚动位置、展开状态。
⑩ 与 QSortFilterProxyModel 联用时特别注意
proxyModel->mapToSource(proxyIndex) 才能拿到源 Model 的 index。
proxyModel->mapFromSource(sourceIndex) 才有对应的代理层 index。
在自定义 Model 的 data() 中接收到的 index 始终是 源 Model 的 index（代理层透明）。

🏆 最佳实践

  1. 数据与 Model 零拷贝：让 Model 直接引用你的数据结构，避免 QStandardItemModel 的双向同步。
  2. 用 QVector / std::vector 存数据：O(1) 随机访问，适合 data() 频繁按行索引的场景。
  3. RAII 包装 begin/end 对：写一个 RowInsertionGuard 类，构造调 begin，析构调 end。
  4. 自定义 role：用 Qt::UserRole + 1、Qt::UserRole + 2 传递自定义数据，不要挤在 DisplayRole。
  5. 配合 QStyledItemDelegate：用 delegate 渲染复杂视觉效果（进度条、星级、图表），Model 只负责数据。
  6. 配合 QSortFilterProxyModel：Model 只存原始数据，过滤和排序交给代理层。
  7. 尽量用具体的子类：写列表就继承 ListModel，写表格就继承 TableModel，不要什么都从 ItemModel 开始。
  8. 单元测试友好：自定义 Model 不依赖任何 View，可以直接在单元测试中实例化和调用 data()。
